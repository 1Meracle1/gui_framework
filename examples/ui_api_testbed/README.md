# UI API Testbed Hot Reload Boundary

Debug builds run `ui_api_testbed.exe` as a stable host and load `ui_api_testbed_module.dll` as the reloadable app module.

Reloadable files watched by the debug host are generated by CMake from the
reloadable module inputs:

- `examples/ui_api_testbed/app.cpp`
- `examples/ui_api_testbed/app.h`
- `examples/assets/ui_api_testbed_texture.png`

Changing one of those files makes the running host execute:

```bat
cmake --build <build-dir> --config Debug --target ui_api_testbed_module --parallel
```

If the build succeeds, the host copies and loads the new DLL, then destroys the previous module runtime.

Files that are intentionally outside the reloadable module require rebuilding
and restarting the host:

- `examples/ui_api_testbed/host.cpp`
- `examples/ui_api_testbed/shared.h`
- `examples/ui_api_testbed/trace.*`
- `src/gui/include/gui/hot_reload.h`
- `src/gui/include/gui/hot_reload_app.h`
- `src/gui/include/gui/hot_reload_overlay.h`
- `src/gui/src/hot_reload.cpp`
- `src/gui/src/hot_reload_app.cpp`
- `src/gui/src/hot_reload_overlay.cpp`
- `CMakeLists.txt`
- `examples/ui_api_testbed/CMakeLists.txt`

The module owns UI state, theme setup, textures, font/cache/draw runtime, and frame command generation. The host owns the process, Win32 window, input, render context/window, presentation loop, and tracing. Generic rebuild/reload control lives in `gui::HotReloadModule`, the app-facing wrapper lives in `gui::HotReloadAppModule`, and the debug status UI lives in `gui::HotReloadOverlay`.

Reusable client hot reload now follows this shape:

- App-specific APIs use `gui::HotReloadAppApi`, or embed `gui::HotReloadApi`
  as their first field when they need extra entry points.
- The exported function returns the full app API pointer.
- The host fills `gui::HotReloadDesc`, calls `gui::init_hot_reload_app_module`,
  `gui::load_hot_reload_app_module`, `gui::update_hot_reload_app_module`, and
  `gui::destroy_hot_reload_app_module`, then calls
  `gui::hot_reload_app_module_api` and `gui::hot_reload_app_module_storage` for
  frame rendering.
- Debug hosts can pass `gui::hot_reload_app_module_status` to
  `gui::HotReloadOverlay` for reusable rebuild output UI.
- CMake code calls `gui_write_hot_reload_manifest` with the reloadable module
  inputs to generate `HOT_RELOAD_WATCH_FILES`.
- Platform details such as `HMODULE`, `FILETIME`, pipes, `CreateProcessW`, and
  DLL copy/unload paths stay inside `src/gui/src/hot_reload.cpp`.

Hot reload is compiled only for Windows debug builds (`BASE_DEBUG`). Release builds link the app module code directly and do not load or rebuild DLLs.
